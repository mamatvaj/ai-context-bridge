01 Project Summary

What we built, why, and the core constraints.

What It Is

A Chrome Extension (Manifest V3) that lets users send selected text or full page content from any webpage directly into the chat interfaces of Claude, ChatGPT, Gemini, Meta AI, and Grok — without APIs, without copy-paste, and without leaving the source page.

Core Constraints

ConstraintImpact
No API keysAll injection must be done via DOM manipulation on the AI platform's own webpage
Manifest V3Service Worker background (not persistent), strict CSP rules, no remote code
5 AI platformsEach uses a completely different editor framework — required 5 separate injectors
Shadow DOM isolationWidget CSS must not be polluted by host page styles — resolved with Shadow DOM
No storage serverAll data (settings, custom prompts) stored in chrome.storage.local only

Final Feature Set

⚡ Floating Widget
Draggable panel with segmented mode toggle, platform selector with SVG icons, preview box, 3 action buttons, custom prompt pills. Lives in Shadow DOM.
💬 Selection Tooltip
Mini quick-action bar appears above selected text. Explain, Summarize, More… buttons. 120ms debounce.
🖱 Right-Click Menu
Chrome context menu with built-in actions + custom prompt templates. Rebuilds dynamically when prompts change via storage.onChanged.
🎯 Custom Prompts
Up to 10 saved templates (name + body). Available in widget, tooltip panel, and right-click menu simultaneously.
🚀 Auto-Submit
Automatically clicks send after injection. Platform-specific strategies per injector.
⚙️ Settings Popup
Real-time saving, no Save button. Default platform, toggles, custom prompt manager, help button → local HTML guide.

04 Bug Log — Widget & CSS

CRITICAL
Widget disappears after page load / navigation
content.js · All pages
Symptom
FAB button and panel appear briefly then vanish completely on React/Next.js apps and SPAs
Root Cause
Widget was appended to document.body. SPA frameworks own the body element and replace its children during route changes and re-renders
Fix
Changed append target to document.documentElement (the <html> element). Frameworks never touch this. Added MutationObserver watching html's direct children to re-inject if widget is removed
Final Fix
Moved entire widget into a Shadow DOM — zero-size host element attached to body. Shadow DOM provides complete CSS isolation and survives framework re-renders
CRITICAL
Page CSS breaks the widget appearance
content.js · widget.css
Symptom
Widget shows on some pages but is invisible, misaligned, or unstyled on others. Especially bad on sites using * { all: unset } or position: static !important
Root Cause
widget.css was injected as a page-level stylesheet via manifest content_scripts. Host page CSS reset rules override our styles — position: fixed, display: flex, z-index all wiped
Fix
All widget CSS moved inside a <style> tag injected into the Shadow DOM. Shadow DOM creates a hard CSS boundary — host page styles cannot pierce it and widget styles cannot leak out. widget.css in manifest removed
MAJOR
Custom Ask expands widget to huge height, buttons stack vertically
content.js · widget panel
Symptom
Clicking Custom Ask caused the textarea to appear AND the action grid to switch to 1-column, making all three buttons stack on top of each other. Panel doubled in height
Root Cause
Textarea was inside .platform-wrap. On show, it pushed the action grid down. The grid had a .custom-open { grid-template-columns: 1fr } override that collapsed it to 1 column
Fix
Moved textarea into its own .custom-wrap.hidden div between platform selector and action buttons. Grid stays permanently 1fr 1fr 1fr. Custom Ask button gets .send-mode class (colour change only) instead of reflowing the grid

05 Bug Log — Settings & Storage

CRITICAL
Always defaulting to Claude — platform selection not saving
popup/popup.js · background.js · content.js
Symptom
User changes default platform to ChatGPT, saves, reopens popup — it shows Claude again. Widget also always opens Claude
Root Cause
Settings were saved/loaded via background service worker messages (AICB_GET_SETTINGS / AICB_SAVE_SETTINGS). MV3 service workers go dormant after ~30 seconds of inactivity. When the worker was asleep, save messages were silently dropped
Fix
Removed the message relay entirely. Popup uses chrome.storage.local.get/set directly. Content script uses chrome.storage.local.get directly. Background worker only handles tab creation and prompt routing
Second issue
Content script was also reading provider from a stale JS variable. Fixed by reading from the DOM select element at send time: shadow.querySelector("#aicb-provider")?.value
MAJOR
Widget provider dropdown not reflecting saved setting on page load
content.js · init()
Symptom
Popup shows correct default platform but widget panel always shows Claude in the dropdown when first opened
Root Cause
Widget loaded settings via background message (same dormant worker issue). Storage read happened before the Shadow DOM select element existed
Fix
chrome.storage.local.get called directly in init() after Shadow DOM and select element are built. Settings also pushed live via AICB_SETTINGS_UPDATED message from popup on every save

06 Bug Log — Text Injection

Each AI platform uses a completely different editor framework. Each required its own investigation and fix strategy.

Gemini

CRITICAL
Only the first line of the prompt is injected into Gemini
injectors/gemini.js
Symptom
Gemini receives only "Please explain the following selected text clearly and concisely:" — the actual selected text is missing
Root Cause
Gemini uses a Quill-based rich-textarea editor. The original code used a ClipboardEvent with a DataTransfer object. Chrome extension content scripts cannot fully construct a trusted DataTransfer — the clipboard data gets sandboxed/stripped. Quill's paste handler received empty/partial data and only committed the first paragraph before giving up. The verification check editor.textContent.length > 0 returned true (first line landed), so the retry loop stopped
Fix
Replaced ClipboardEvent approach entirely. Use document.execCommand("insertText", false, fullText) — this inserts the complete multi-line string atomically, bypasses Quill's clipboard interceptor, and works reliably from extension content scripts. New verification checks the tail of the text, not just any text presence
MAJOR
Gemini Auto-Submit not firing
injectors/gemini.js
Symptom
Text injects correctly but auto-submit does nothing — send button stays disabled
Root Cause 1
Angular events dispatched only to the inner .ql-editor div. Gemini's Angular component model lives on the <rich-textarea> host element — that's where zone.js binds its listeners. The button's disabled state is computed by the Angular component, which never saw the events
Root Cause 2
Send button disabled check used button.disabled (DOM property) only. Angular's template binding can set [disabled] attribute while the property reads false — inconsistent state
Root Cause 3
Gemini's send button has no stable data-test-id or aria-label in the chat UI — confirmed by HAR analysis of Gemini's minified Angular bundle
Fix
Events now fired on BOTH .ql-editor AND rich-textarea host. Send button found by structural DOM walking from editor upward (not by attribute selector). Button disabled check uses both btn.disabled AND btn.hasAttribute("disabled"). Enter key dispatched on host element as primary submit method; structural button click as fallback

Meta AI

CRITICAL
Nothing transfers to Meta AI — just redirects to the site
injectors/meta.js
Symptom
Tab opens on meta.ai but the input box is completely empty
Root Cause
All previous attempts targeted div[contenteditable='true'] — Lexical editor assumptions. Meta AI does not use a contenteditable div. Confirmed via HAR analysis: Meta's own Vercel Speed Insights telemetry explicitly logged the element that received keyboard events as: input.text-subheadline.placeholder:text-text-tertiary.min-w-0.flex-1.bg-transparent.px-2.py-3.outline-none — a plain HTML <input type="text">. In conversation view it becomes a <textarea>
Fix
Rewrote injector to target input.text-subheadline and textarea[class*='resize-none']. Used React native prototype setter: Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, "value").set.call(field, text) — the only method that triggers React's synthetic onChange on a controlled input
CRITICAL
Text appears briefly then vanishes (flicker)
injectors/meta.js
Symptom
Text visibly types into the input box, then disappears after ~100ms leaving it empty again
Root Cause
Meta AI's input is a React controlled component — React owns the value via useState. After our injection, React reconciles its virtual DOM. A competing state update (likely a useEffect or concurrent re-render) fires with the old empty state and React commits it — overwriting our injected value
Root Cause 2
The MutationObserver was set with subtree: true — it fired on every React re-render (dozens/second), causing the tryInject() function to be called 50+ times simultaneously before injected = true was set inside an async callback
Fix 1 — Race condition
Added synchronous inProgress flag set BEFORE the async chrome.storage.local.get() call. All concurrent calls blocked at gate. Observer killed immediately on success before React re-renders begin
Fix 2 — Observer
Changed subtree: true → subtree: false + 300ms debounce. Fires only on direct body children (catches SPA navigation) not every inner DOM mutation
Fix 3 — Hold loop
After injection, a setInterval checks every 100ms if React reset the value to empty. If so, re-applies and fires events. Stops when value is stable or after 2 seconds. Cancelled before submit fires
MAJOR
execCommand("selectAll") deletes entire page DOM
injectors/meta.js (earlier versions)
Symptom
After injection attempt, the AI platform page content disappears
Root Cause
document.execCommand("selectAll") is a page-level command. It selects all content on the entire document, not just the focused input. The subsequent execCommand("delete") then deletes the entire page DOM
Fix
Replaced with Range API: range.selectNodeContents(editor) which scopes the selection precisely to the editor's contents only. document.execCommand("delete") then only clears the input field

Grok

MAJOR
Nothing transfers to Grok
injectors/grok.js
Symptom
Tab opens on grok.com but text box is empty
Root Cause
Grok uses a React-controlled textarea. Direct assignment editor.value = text does not trigger React's internal state update. React's controlled component ignores values set this way on next re-render
Fix
Applied same React native prototype setter used for Meta AI. Added input + change events after setter. Also improved selector order to target textarea[placeholder*='Ask'] before generic fallbacks

07 Bug Log — Auto-Submit

CRITICAL

background.js · all injectors
Symptom

Root Cause

08 Deep Dive — Shadow DOM Architecture

Understanding why Shadow DOM was necessary and how it's structured is critical for any future widget work.

The Problem With Regular Injection

When a content script appends elements to the page, those elements are subject to all CSS rules on that page. Any site using a CSS reset like * { all: unset } or * { box-sizing: content-box !important } will break our widget's layout. This affects a surprisingly large number of sites — news sites, documentation, React apps.

How Our Shadow DOM Works

// Structure in content.js: const host = document.createElement("div"); host.id = "aicb-host"; host.style.cssText = "all:initial;position:fixed;top:0;left:0;width:0;height:0;..."; // Zero-size: no layout impact on host page whatsoever shadow = host.attachShadow({ mode: "open" }); // "open" allows chrome.devtools to inspect it (useful for debugging) const style = document.createElement("style"); style.textContent = WIDGET_CSS; // All CSS lives inside shadow — can't be touched by page shadow.appendChild(style); // All widget elements appended to shadow, not document.body shadow.appendChild(fab); shadow.appendChild(panel); shadow.appendChild(tip);

Key Shadow DOM Rules to Remember

RuleImplication
document.querySelector()Does NOT find elements inside shadow DOM. Always use shadow.querySelector() for widget elements
Event bubblingEvents from inside shadow DOM bubble to the host element but carry composedPath() — use this to check if clicks originated from our widget
CSS variablesCSS custom properties DO pierce shadow boundaries — but only if defined on :host or the host element itself
FontsWeb fonts loaded in the page ARE available inside shadow DOM
Shadow mode "open"Accessible via element.shadowRoot from JavaScript. "closed" would block this
💡
When debugging the widget in DevTools, you'll see #shadow-root (open) inside #aicb-host. All widget elements are inside this shadow root.

09 Deep Dive — Race Condition in Injectors

The Pattern That Caused It

This pattern was the source of the Meta AI flicker bug and instability in Gemini. It's a subtle async race that's easy to reintroduce:

// ❌ BROKEN — original pattern let injected = false; function tryInject() { if (injected) return; // ← checked synchronously ✓ chrome.storage.local.get(..., (res) => { // ← by the time this callback runs, tryInject() has been // called 50 more times. They all passed the injected check. // All 50 are now in this callback simultaneously. injected = true; // ← set too late, inside async injectText(editor, text); // ← runs 50 times, fighting React }); } observer.observe(body, { childList: true, subtree: true }); // ← fires 100/sec on React
// ✅ FIXED — current pattern let injected = false; let inProgress = false; // ← NEW: synchronous lock function tryInject() { if (injected || inProgress) return; // ← both checked synchronously inProgress = true; // ← SET synchronously before async chrome.storage.local.get(..., (res) => { if (injected) { inProgress = false; return; } // ← re-check after await if (observer) { observer.disconnect(); } // ← kill observer BEFORE inject injected = true; inProgress = false; injectText(editor, text); }); } let debounce; observer = new MutationObserver(() => { if (injected || inProgress) return; clearTimeout(debounce); debounce = setTimeout(tryInject, 300); // ← debounced, not immediate }); observer.observe(body, { childList: true, subtree: false }); // ← subtree FALSE
⚠️
If you ever modify an injector, do not change subtree: false to subtree: true without also having the inProgress lock and debounce. React apps on Meta/Gemini will instantly reintroduce the flicker bug.

10 Deep Dive — HAR File Analysis

When guessing at DOM selectors stopped working, we recorded HAR (HTTP Archive) files from the AI platforms and analysed the JavaScript bundles and telemetry payloads inside them.

What We Found for Meta AI

Meta's Vercel Speed Insights telemetry in the HAR contained this payload:

{ "type": "FID", "attribution": { "eventTarget": "input.text-subheadline.placeholder:text-text-tertiary.min-w-0.flex-1.bg-transparent.px-2.py-3.outline-none", "eventType": "keydown" } }

This revealed that Meta AI's chat input was a plain <input> element, not a contenteditable div. Every previous injection approach — Lexical beforeinput events, ClipboardEvent paste, contenteditable manipulation — was targeting the wrong element type entirely.

What We Found for Gemini

Gemini's minified Angular bundle revealed that the send button is rendered by an Angular component with no stable data-test-id or aria-label in the chat UI. The submit handler was a minified method b.Gj() bound via Angular's click binding — impossible to call directly. The only reliable approach was structural DOM walking from the editor element upward to find the send button by proximity.

How to Capture a HAR File

1. Open Chrome DevTools (F12) 2. Go to Network tab 3. Check "Preserve log" 4. Navigate to the AI platform and interact with it 5. Right-click any request → "Save all as HAR with content" 6. Analyse with Python: import json, re with open("platform.har", "r", errors="replace") as f: content = f.read() # Search for editor/button selectors matches = re.findall(r'aria-label[^"]{0,5}"[^"]{0,40}[Ss]end[^"]{0,40}"', content) print(matches)

Key Things to Know Before Touching Injectors

🚨
Never change MutationObserver subtree: false to subtree: true without the inProgress lock + debounce. React platforms will instantly get the flicker bug back.
🚨
Never use document.execCommand("selectAll") inside an injector. It selects the entire page, not just the editor. Use range.selectNodeContents(editor) instead.
⚠️
The autoSubmit flag must be passed through pendingPrompt in background.js → storeAndOpen(). If you reset or rewrite that function, ensure autoSubmit is always included.
💡
When a platform's injection breaks, first check the HAR. Record a new HAR from that platform and search for contenteditable, aria-label.*[Ss]end, and data-test-id. This reveals the actual DOM selectors faster than guessing.
